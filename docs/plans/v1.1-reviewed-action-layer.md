# v1.1 Reviewed Action Layer Baseline

**Goal:** Keep the v1.0 read-only governance pipeline intact while adding a v1.1 reviewed/proposal layer that is safe by default, locally testable, and clearly separated under `artifacts/v1.1/`.

**Current behavior:** v1.1 generates and validates local proposal, template, adapter, provenance, evidence, smoke, leak-scan, and pipeline reports. It does not provide an apply command and does not execute reviewed actions.

## Safety constraints

- Proposal-only by default.
- Reviewed apply remains gated; no apply execution command is implemented in v1.1.
- `destructive_operations_allowed` is false by default.
- `remote_publication_allowed` is false by default.
- Raw archive apply is blocked.
- Delete, move, quarantine, and rehydrate operations are blocked.
- Graph mutation and graph/events mutation are blocked.
- Sensitive export is blocked.
- `provenance append --local-test` writes only local artifact JSON/JSONL under `artifacts/v1.1/provenance/`.
- Before any commit, verify `git var GIT_AUTHOR_IDENT` and `git var GIT_COMMITTER_IDENT`; both must be `VANILAAAAAAAA <xchen247@uw.edu>`.

## Artifact layout

- v1 pipeline artifacts remain under `artifacts/v1/`.
- v1.1 pipeline artifacts are generated under `artifacts/v1.1/`.
- v1.1 proposal manifests use deterministic default paths:
  - `artifacts/v1.1/proposals/reviewed-apply-plan.json`
  - `artifacts/v1.1/proposal-manifest.schema.json`
  - `artifacts/v1.1/proposal-validation.json`
- v1.1 adapter reports use:
  - `artifacts/v1.1/adapter-report.json`
  - `artifacts/v1.1/adapter-report.md`
- v1.1 provenance local-test reports use:
  - `artifacts/v1.1/provenance/local-test-events.jsonl`
  - `artifacts/v1.1/provenance/local-append-report.json`
  - `artifacts/v1.1/provenance/current-state.json`

Generated artifacts are local validation outputs and should not be committed unless explicitly promoted as documentation/templates.

## Proposal manifest expectations

A v1.1 proposal manifest is reviewable JSON with stable safety fields, including:

- `proposal_id`
- `created_at`
- `version`
- `scope`
- `operation_type`
- `target`
- `reason`
- `evidence`
- `risk_level`
- `required_approvals`
- `destructive`
- `apply_allowed`
- `blockers`
- `warnings`

Validation fails if required fields are missing, evidence is empty, destructive behavior is requested, apply is allowed without review, remote publication is requested, or blocked action types are present.

## Adapter report expectations

Adapter reports are read-only. They document adapter name, capabilities, limitations, inputs, outputs, safety boundary, proposal hints, warnings, and blockers. Failure modes are represented as warnings/blockers or gated proposal notes; adapters do not execute side effects.
